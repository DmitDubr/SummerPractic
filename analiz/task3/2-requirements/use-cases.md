# Use Cases — картинг-центр «Апекс»

> Источник: [domain-description.md](../1-elicitation/domain-description.md).
> В скоупе текущей поставки — актор **Клиент** (R-028). Действия владельца и маршала выполняются в существующей инфраструктуре.
> Приоритет: **Must** — MVP v1; **Should** — v2.

---

## UC-001. Просмотр расписания заездов

| Поле | Описание |
| :-- | :-- |
| **ID** | UC-001 |
| **Название** | Просмотр расписания заездов |
| **Актор** | Клиент |
| **Приоритет** | Must |
| **Источник** | [domain-description.md §5.1](../1-elicitation/domain-description.md#51-просмотр-расписания) |
| **Связь** | FR-001–FR-005; US-001–US-005 |

**Предусловия:** Клиент открыл клиентское приложение. Бэкенд доступен и отдаёт расписание.

**Основной сценарий:**
1. Система запрашивает слоты на ближайшие 7 дней (R-027).
2. Система отображает список заездов: время, конфигурация трассы (кратко), маршал, «есть места / мест нет», цена.
3. Клиент при необходимости применяет фильтры (время суток, маршал) или расширяет период через фильтр дат.
4. Клиент выбирает интересующий слот для дальнейших действий.

**Альтернативные сценарии:**
- **2a.** Слотов на ближайшие дни нет → система показывает empty state «Пока нет доступных заездов».

## UC-002. Запись на заезд

| Поле | Описание |
| :-- | :-- |
| **ID** | UC-002 |
| **Название** | Запись на заезд |
| **Актор** | Клиент |
| **Приоритет** | Must |
| **Источник** | [domain-description.md §5.2](../1-elicitation/domain-description.md#52-бронирование) |
| **Связь** | FR-006–FR-013; US-006–US-008, US-019 |

**Предусловия:** Клиент просматривает расписание (UC-001). В слоте есть свободные места. Прокатный фонд доступен (если выбран прокат).

**Основной сценарий:**
1. Клиент выбирает слот.
2. Клиент указывает имя и телефон (при первой записи).
3. Клиент при необходимости указывает **участников** в одной брони.
4. Клиент выбирает: своя экипировка или прокат (шлем, подшлемник).
5. Система отправляет запрос бронирования в бэкенд.
6. Бэкенд атомарно проверяет наличие картов и проката, подтверждает бронь (R-004).
7. Система отображает клиенту подтверждение записи и цену (оплата на месте).

**Альтернативные сценарии:**
- **4a.** Прокатный фонд исчерпан → слот **недоступен** для записи.
- **6a.** Мест нет → бэкенд отклоняет запрос → система информирует клиента об отказе.
- **6b.** Заезд заполнен → система отображает «мест нет».

## UC-003. Просмотр своих записей

| Поле | Описание |
| :-- | :-- |
| **ID** | UC-003 |
| **Название** | Просмотр своих записей |
| **Актор** | Клиент |
| **Приоритет** | Must |
| **Источник** | [domain-description.md §2, §5.3](../1-elicitation/domain-description.md#2-ключевые-сущности) |
| **Связь** | FR-014; US-009 |

**Предусловия:** У клиента есть одна или несколько броней в системе.

**Основной сценарий:**
1. Клиент открывает раздел своих записей.
2. Система отображает список броней со статусами (активна, отменена клиентом, отменена центром, посещена и др.).
3. Клиент просматривает детали выбранной записи.

**Альтернативные сценарии:**
- **2a.** Нет подключения к интернету → система показывает последний закэшированный список записей (NFR-009).

## UC-004. Отмена записи клиентом

| Поле | Описание |
| :-- | :-- |
| **ID** | UC-004 |
| **Название** | Отмена записи клиентом |
| **Актор** | Клиент |
| **Приоритет** | Must |
| **Источник** | [domain-description.md §3, §5.3](../1-elicitation/domain-description.md#53-отмена-брони) |
| **Связь** | FR-015, FR-016; US-010, US-011 |

**Предусловия:** У клиента есть активная бронь на предстоящий заезд.

**Основной сценарий:**
1. Клиент открывает активную бронь.
2. Клиент инициирует отмену.
3. Если до начала ≥ 1 ч — система отменяет бронь и освобождает карт в слоте.
4. Система обновляет статус брони на «отменена клиентом».

**Альтернативные сценарии:**
- **3a.** До начала < 1 ч → система показывает **предупреждение**; клиент подтверждает отмену → бронь отменяется; на стороне админки фиксируется **пометка** (штрафов в MVP нет).

## UC-005. Отмена заезда центром (форс-мажор, погода)

| Поле | Описание |
| :-- | :-- |
| **ID** | UC-005 |
| **Название** | Отмена заезда центром (форс-мажор, погода) |
| **Актор** | Клиент (получатель); инициатор — существующая инфраструктура (вне скоупа) |
| **Приоритет** | Must |
| **Источник** | [domain-description.md §3, §5.6](../1-elicitation/domain-description.md#56-отмена-по-погоде) |
| **Связь** | FR-017, FR-018; US-012, US-013 |

**Предусловия:** Клиент записан на слот. Владелец/администратор отменяет заезд в существующей инфраструктуре (погода, поломка, нехватка маршала и т. п.).

**Основной сценарий:**
1. Бэкенд переводит бронь в статус **«Отменён центром»** с указанием причины (R-008).
2. Клиентское приложение получает обновление статуса.
3. Клиент видит в приложении статус и причину отмены.

**Альтернативные сценарии:**
- **3a.** (v2) Система отправляет клиенту **push** с предложением перезаписаться на другой заезд → клиент переходит к выбору слота из push (FR-023, FR-024).

**Постусловия:** Повторная запись на тот же отменённый слот запрещена (R-008).

## UC-006. Перенос заезда

| Поле | Описание |
| :-- | :-- |
| **ID** | UC-006 |
| **Название** | Уведомление о переносе заезда |
| **Актор** | Клиент (получатель); инициатор — существующая инфраструктура (вне скоупа) |
| **Приоритет** | Should |
| **Источник** | [domain-description.md §3](../1-elicitation/domain-description.md#отмена-центром-и-погода-r-008) |
| **Связь** | FR-025; US-014 |

**Предусловия:** Клиент записан на слот. Администратор изменяет время или маршала без полной отмены заезда.

1. Бэкенд обновляет данные слота (время и/или маршал).
2. Система отправляет клиенту уведомление (**push и SMS**) о переносе.
3. Клиент видит актуальные данные в детали записи.

## UC-007. Оценка маршала

| Поле | Описание |
| :-- | :-- |
| **ID** | UC-007 |
| **Название** | Оценка маршала после заезда |
| **Актор** | Клиент |
| **Приоритет** | Should |
| **Источник** | [domain-description.md §3, §5.5](../1-elicitation/domain-description.md#55-оценка-маршала-v2) |
| **Связь** | FR-026–FR-028; US-015, US-016 |

**Предусловия:** Клиент посетил заезд; бронь имеет статус «посещена». С момента заезда прошло не более недели.

**Основной сценарий:**
1. Система предлагает клиенту оценить маршала (звёзды).
2. Клиент выставляет или изменяет оценку.
3. Система сохраняет оценку через API; рейтинг отображается другим клиентам при выборе слота.

**Постусловия:** У клиента — не более одной оценки на маршала.

## UC-008. Уведомления клиенту

| Поле | Описание |
| :-- | :-- |
| **ID** | UC-008 |
| **Название** | Уведомления клиенту |
| **Актор** | Клиент |
| **Приоритет** | Should |
| **Источник** | [domain-description.md §3, §5.4](../1-elicitation/domain-description.md#54-напоминания-v2) |
| **Связь** | FR-029; US-017 |

**Предусловия:** У клиента есть активная бронь или произошло событие отмены.

**Основной сценарий:**
1. Система формирует уведомление по событию: напоминание **за 2 часа**, отмена (клиент / центром, в т.ч. погода).
2. Клиент получает **push и SMS**.

**Примечания:** Доставка через push и SMS (NFR-010).

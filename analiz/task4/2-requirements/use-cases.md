# Use Cases — гончарная мастерская «Глина»

> Источник: [domain-description.md](../1-elicitation/domain-description.md).
> В скоупе текущей поставки — актор **Клиент** (R-028). Действия владельца и мастера выполняются в существующей инфраструктуре.
> Приоритет: **Must** / **Should**.

---

## UC-001. Просмотр расписания занятий

| Поле | Описание |
| :-- | :-- |
| **ID** | UC-001 |
| **Название** | Просмотр расписания занятий |
| **Актор** | Клиент |
| **Приоритет** | Must |
| **Источник** | [domain-description.md §5.1](../1-elicitation/domain-description.md#51-просмотр-расписания) |
| **Связь** | FR-001–FR-005; US-001–US-005 |

**Предусловия:** Клиент открыл клиентское приложение. Бэкенд доступен и отдаёт расписание.

**Основной сценарий:**
1. Система запрашивает слоты на ближайшие 7 дней (R-027).
2. Система отображает список занятий: время, программа (кратко), мастер, рейтинг, «есть места / мест нет», цена.
3. Клиент при необходимости применяет фильтры (время суток, уровень, программа) или расширяет период через фильтр дат.
4. Клиент выбирает интересующий слот для дальнейших действий.

**Альтернативные сценарии:**
- **2a.** Слотов на ближайшие дни нет → система показывает empty state «Пока нет доступных занятий».

## UC-002. Запись на мастер-класс

| Поле | Описание |
| :-- | :-- |
| **ID** | UC-002 |
| **Название** | Запись на мастер-класс |
| **Актор** | Клиент |
| **Приоритет** | Must |
| **Источник** | [domain-description.md §5.2](../1-elicitation/domain-description.md#52-бронирование) |
| **Связь** | FR-006–FR-012, FR-025; US-006–US-008, US-018, US-019 |

**Предусловия:** Клиент просматривает расписание (UC-001). В слоте есть свободные места. У клиента нет другой записи на этот день.

**Основной сценарий:**
1. Клиент выбирает слот.
2. Клиент указывает имя и телефон (при первой записи).
3. Клиент выбирает: своё снаряжение или прокат (инструменты, фартук).
4. Система отправляет запрос бронирования в бэкенд.
5. Бэкенд атомарно проверяет наличие мест и подтверждает бронь (R-004).
6. Система отображает клиенту подтверждение записи и цену (оплата на месте).

**Альтернативные сценарии:**
- **3a.** Прокатный фонд исчерпан → запись возможна только «со своим» снаряжением.
- **5a.** Мест нет или уже есть запись на этот день → бэкенд отклоняет запрос → система информирует клиента об отказе.
- **5b.** Группа заполнена → система отображает «мест нет».

## UC-003. Просмотр своих записей

| Поле | Описание |
| :-- | :-- |
| **ID** | UC-003 |
| **Название** | Просмотр своих записей |
| **Актор** | Клиент |
| **Приоритет** | Must |
| **Источник** | [domain-description.md §2, §5.3](../1-elicitation/domain-description.md#2-ключевые-сущности) |
| **Связь** | FR-013; US-009 |

**Предусловия:** У клиента есть одна или несколько броней в системе.

**Основной сценарий:**
1. Клиент открывает раздел своих записей.
2. Система отображает список броней со статусами (активна, отменена клиентом, отменена мастерской, посещена и др.).
3. Клиент просматривает детали выбранной записи.

**Альтернативные сценарии:**
- **2a.** Нет подключения к интернету → система показывает последний закэшированный список записей (NFR-009).

## UC-004. Отмена записи клиентом

| Поле | Описание |
| :-- | :-- |
| **ID** | UC-004 |
| **Название** | Отмена записи клиентом |
| **Актор** | Клиент |
| **Приоритет** | Must |
| **Источник** | [domain-description.md §3, §5.3](../1-elicitation/domain-description.md#53-отмена-брони) |
| **Связь** | FR-014, FR-015; US-010, US-011 |

**Предусловия:** У клиента есть активная бронь на предстоящее занятие.

**Основной сценарий:**
1. Клиент открывает активную бронь.
2. Клиент инициирует отмену.
3. Если до начала ≥ 3 ч — система отменяет бронь и освобождает место в слоте.
4. Система обновляет статус брони на «отменена клиентом».

**Альтернативные сценарии:**
- **3a.** До начала < 3 ч → система показывает предупреждение о заготовленной глине; клиент подтверждает отмену → бронь отменяется (штрафов в MVP нет).

## UC-005. Отмена занятия мастерской (форс-мажор)

| Поле | Описание |
| :-- | :-- |
| **ID** | UC-005 |
| **Название** | Отмена занятия мастерской (форс-мажор) |
| **Актор** | Клиент (получатель); инициатор — существующая инфраструктура (вне скоупа) |
| **Приоритет** | Must |
| **Источник** | [domain-description.md §3, §5.5](../1-elicitation/domain-description.md#55-форс-мажор) |
| **Связь** | FR-016–FR-019; US-012, US-013 |

**Предусловия:** Клиент записан на слот. Владелец/администратор отменяет занятие в существующей инфраструктуре (поломка печи, отключение электричества и т. п.).

**Основной сценарий:**
1. Бэкенд переводит бронь в статус «Отменено мастерской» с указанием причины (R-008).
2. Клиентское приложение получает обновление статуса.
3. Система отправляет клиенту push-уведомление с просьбой перезаписаться на другое занятие.
4. Клиент видит в приложении статус, причину и может перейти к перезаписи.

**Постусловия:** Повторная запись на тот же отменённый слот запрещена (R-008).

| Поле | Описание |
| :-- | :-- |
| **ID** | UC-006 |
| **Название** | Уведомление о переносе занятия |
| **Актор** | Клиент (получатель); инициатор — существующая инфраструктура (вне скоупа) |
| **Приоритет** | Must |

| **Связь** | FR-020; US-014 |

**Предусловия:** Клиент записан на слот. Администратор изменяет время или мастера без полной отмены занятия.

1. Бэкенд обновляет данные слота (время и/или мастер).
2. Система отправляет клиенту push-уведомление о переносе.
3. Клиент видит актуальные данные в детали записи.

## UC-007. Оценка мастера

| Поле | Описание |
| :-- | :-- |
| **ID** | UC-007 |
| **Название** | Оценка мастера после занятия |
| **Актор** | Клиент |
| **Приоритет** | Must |
| **Источник** | [domain-description.md §3, §5.4](../1-elicitation/domain-description.md#54-оценка-мастера) |
| **Связь** | FR-021–FR-023; US-015, US-016 |

**Предусловия:** Клиент посетил занятие; бронь имеет статус «посещена». С момента занятия прошло не более недели.

**Основной сценарий:**
1. Система предлагает клиенту оценить мастера (звёзды).
2. Клиент выставляет или изменяет оценку.
3. Система сохраняет оценку через API; рейтинг отображается другим клиентам при выборе слота.

**Постусловия:** У клиента — не более одной оценки на мастера.

## UC-008. Push-уведомления клиенту

| Поле | Описание |
| :-- | :-- |
| **ID** | UC-008 |
| **Название** | Push-уведомления клиенту |
| **Актор** | Клиент |
| **Приоритет** | Must |
| **Источник** | [domain-description.md §3](../1-elicitation/domain-description.md#уведомления-mvp) |
| **Связь** | FR-024; US-017 |

**Предусловия:** У клиента есть активная бронь или завершённый сценарий записи/отмены.

**Основной сценарий:**
1. Система формирует push-уведомление по событию: напоминание за день, за 2 часа, подтверждение записи, отмена (клиент / мастерская), перенос занятия.
2. Клиент получает push в приложении.

**Примечания:** Доставка только через push (NFR-010).

## Обзор

Собрать **имя и телефон** клиента при **первой записи** — этого достаточно для идентификации без полноценной регистрации. Для возвращающихся клиентов поля предзаполняются. Бейдж **«Постоянный клиент»** отображается при флаге `isRegularClient` с бэкенда; при записи применяется **приоритет** (`hasPriorityBooking`, FR-025), скидок в MVP нет. Отдельный таб «Профиль» **не создаётся**.

Секция встроена в поток бронирования на SCR-005: **контакты (SCR-012) → экипировка → подтверждение**.

### User Story

> Как клиент, я хочу указать имя и телефон при записи на занятие, чтобы мастерская могла меня идентифицировать без регистрации с паролем.

**Не в MVP:** лист ожидания при `NO_SPOTS` (FR-011) — только возврат к расписанию; фильтр по мастеру; iOS; email, пароль, аватар, отдельный экран настроек.

## Входные данные

| Название | Тип | Источник | Описание |
| :-- | :-- | :-- | :-- |

| `client` | `ClientContacts` | out → `createBooking` | `{ name, phone }` при submit |

| Условие | UI |
| :-- | :-- |
| `isComplete = false` или 404 `getProfile` | Inline-поля «Имя» и «Телефон» на SCR-005 |

| :-- | :-- | :-- |

## Используемые запросы

### getProfile

**Метод:** GET  
**Путь:** `/profile`  
**Спецификация:** [../../api/openapi.yaml](../../api/openapi.yaml) → `getProfile`

**Обработка ответа:**

| HTTP / код | UI-реакция |
| :-- | :-- |
| 200 + `isComplete: true` | Сводка контактов + бейдж при `isRegularClient` |
| 200 + `isComplete: false` | Inline-поля на SCR-005 |
| 404 | Inline-поля (первая запись) |
| 401 | Inline-поля; сессия не требуется для первой записи |
| 5xx / timeout | Error на SCR-005 + retry; inline-поля доступны для ручного ввода |

### updateProfile

**Метод:** PATCH  
**Путь:** `/profile`  
**Спецификация:** [../../api/openapi.yaml](../../api/openapi.yaml) → `updateProfile`

**Обработка ответа:**

| HTTP / код | UI-реакция |
| :-- | :-- |
| 200 + data | Sheet закрывается; сводка на SCR-005 обновлена; `sessionToken` сохранён при первом upsert |
| 400 `VALIDATION_ERROR` | Inline-ошибки под полями |
| 5xx / timeout | Снек «Не удалось сохранить» + «Повторить» |

**Сохранение профиля:**

1. **Явный:** `updateProfile` при «Сохранить» в sheet SCR-012.
2. **Неявный:** upsert в составе `createBooking` (api-sequence §4.2).

### createBooking (контакты в составе формы SCR-005)

**Метод:** POST  
**Путь:** `/bookings`  
**Спецификация:** [../../api/openapi.yaml](../../api/openapi.yaml) → `createBooking`

Поле `client: { name, phone }` передаётся при submit. После 201 — сохранить `sessionToken` из ответа.

## Элементы экрана

| Элемент | Описание | Источник данных | Валидация / поведение |
| :-- | :-- | :-- | :-- |
| Заголовок секции | «Контактные данные» или «Как к вам обращаться» | — | Всегда виден |
| Поле «Имя» | Текстовый input, обязательный | `profile.name` | Placeholder «Ваше имя»; 1–50 символов; «Укажите имя» |
| Поле «Телефон» | Маска **+7 (XXX) XXX-XX-XX** | `profile.phone` | Цифровая клавиатура; паттерн `^\+7\d{10}$`; автоподстановка `+7`; нормализация вставки из буфера |
| Бейдж «Постоянный клиент» | Chip рядом с заголовком | `profile.isRegularClient` | **Только отображение** + приоритет на бэкенде (FR-025); без иконки «скидка» |
| Сводка (повторный визит) | Имя + маскированный телефон | `profile.*` | Формат «Иван · +7 900 ***-45-67»; линк «Изменить» |
| Валидация inline | Под полем | Локально | «Укажите имя», «Введите корректный номер»; красная обводка |
| Кнопка «Сохранить» | В bottom sheet | — | Активна при изменениях и валидности; loader при `updateProfile` |
| CTA «Записаться» | На SCR-005 | — | Disabled до валидных контактов и выбора экипировки |

**Не включать:** email, пароль, аватар, отдельный экран настроек, переключатели уведомлений (push — на SCR-006), запрос разрешений контактов/SMS.

**Терминология:** **мастер**, **занятие**, **программа**; прокат (инструменты, фартук) **не влияет на цену** (FR-012). Один человек = один участник занятия; имя в профиле = имя гостя в мастерской.

**Критерии дизайна:**

- Секция — часть формы записи на **занятие в гончарной мастерской**, а не «регистрация в приложении».
- Бейдж «Постоянный клиент» — позитивный, скромный chip; можно нейтральную иконку «⭐» или «🏺» по UI Kit.
- Bottom sheet для правки — единый паттерн с SCR-002, SCR-003.
- Цену и оплату **не дублировать** в SCR-012 — они на SCR-005 (FR-012).
- Доступность: label связан с input, ошибки озвучиваются screen reader, контраст бейджа WCAG AA.

## Состояния экрана

| Состояние | Условие | Отображение |
| :-- | :-- | :-- |
| Loading | `getProfile` в процессе | Skeleton секции контактов |
| **Первая запись, пустой профиль** | 404 или `isComplete=false` | Оба поля пустые; бейдж скрыт (если не regular); CTA disabled до валидных контактов и экипировки |
| **Первая запись, regular client** | `isRegularClient=true` | Бейдж «Постоянный клиент» после загрузки профиля |
| **Повторная запись** | `isComplete=true` | Сводка предзаполнена; поля в основном потоке свёрнуты |
| **Редактирование (sheet)** | Тап «Изменить» | Поля с текущими значениями; «Сохранить» при изменениях и валидности |
| **Валидация ошибки** | Невалидное имя/телефон | Красная обводка + текст ошибки; фокус на первое невалидное |
| Submitting | `updateProfile` / `createBooking` | Loader на кнопке SCR-005 или sheet |
| **Офлайн** | Нет сети, профиль не сохранён | Блокировка новой записи; сообщение на SCR-005 |
| **Миграция** | Имя известно, телефон пустой | Показать только поле «Телефон» |

> Сквозной паттерн — [LOGIC-008](../../5-mobile-app-spec/09_Логики/LOGIC-008_Паттерн-состояний-экрана.md).

## Сценарии и переходы

1. **Первая запись:** клиент выбирает занятие → SCR-005 → секция SCR-012 развёрнута → вводит имя и телефон → экипировка → «Записаться» → `createBooking` с `client` → SCR-006; `sessionToken` сохранён.
2. **Повторная запись:** SCR-005 загружает профиль → сводка «Мария · +7 916 ***-12-34» → клиент может сразу записаться или тапнуть «Изменить» → sheet SCR-012 → правка → «Сохранить» (`updateProfile`) → SCR-005.
3. **Постоянный клиент:** API возвращает `isRegularClient: true` → бейдж «Постоянный клиент» на секции; при submit бэкенд применяет приоритет записи (FR-025); **без изменения цены** и без отдельного UI приоритета.
4. **Перезапись после отмены мастерской:** push → SCR-009 → SCR-001 → SCR-004 → SCR-005; если профиль есть — контакты в сводке SCR-012, далее экипировка.
5. **Краевой случай миграции:** имя известно бэкенду, телефон пустой → показать только поле «Телефон».

## Критерии приёмки

| ID | Критерий |
| :-- | :-- |
| AC-001 | **Дано** пустой профиль (404 `getProfile`), **Когда** открыт SCR-005, **Тогда** inline-секция SCR-012 с обязательными полями «Имя» и «Телефон» и CTA «Записаться» disabled до их валидного заполнения и выбора экипировки. |
| AC-002 | **Дано** пользователь вводит телефон, **Когда** отображается поле, **Тогда** применяется маска **+7 (XXX) XXX-XX-XX** с автоподстановкой `+7` и нормализацией вставки из буфера. |
| AC-003 | **Дано** невалидный телефон (не `^\+7\d{10}$`), **Когда** пользователь пытается продолжить, **Тогда** показывается inline-ошибка «Введите корректный номер» и `createBooking` не отправляется. |
| AC-004 | **Дано** `getProfile` вернул `isComplete=true`, **Когда** открыт SCR-005, **Тогда** показывается сводка «{name} · +7 XXX ***-XX-XX» и ссылка «Изменить» вместо пустых полей. |
| AC-005 | **Дано** `isRegularClient=true`, **Когда** отображается секция контактов, **Тогда** виден бейдж «Постоянный клиент» без UI скидок и изменения цены (FR-025). |
| AC-006 | **Дано** пользователь тапнул «Изменить» на повторной записи, **Когда** правит контакты и нажимает «Сохранить», **Тогда** выполняется `updateProfile`, sheet закрывается и сводка на SCR-005 обновлена. |
| AC-007 | **Дано** первая успешная запись, **Когда** `createBooking` вернул 201, **Тогда** `sessionToken` сохранён локально и профиль доступен для следующих визитов. |
| AC-008 | **Дано** нет сети и профиль не сохранён, **Когда** пользователь на SCR-005, **Тогда** новая запись заблокирована с понятным сообщением об офлайне. |

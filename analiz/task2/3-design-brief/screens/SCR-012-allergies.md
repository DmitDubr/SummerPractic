# Аллергии и ограничения питания

**ID:** SCR-012  
**Тип:** Секция inline / Bottom Sheet (без отдельной вкладки)  
**Домен:** 04. Профиль и аллергии  
**Приоритет:** Critical  
**Статус:** Актуален  
**Сессия клиента:** ClientSession — Bearer после первой записи (для `updateBookingAllergies`; `checkAllergyCompatibility` доступен без сессии)  
**Дизайн-макет:** Figma — TBD · **Design brief:** [SCR-012-allergies.md](SCR-012-allergies.md)

> **Не отдельный таб.** SCR-012 — **обязательный шаг** в потоке записи на [SCR-005](SCR-005-booking-form.md) и **редактируемая секция** на [SCR-009](SCR-009-booking-detail.md) для активной брони. ## Обзор

Собрать информацию об **аллергиях и пищевых ограничениях** клиента **до начала класса**, чтобы кухня студии «Шеф-стол» могла заранее учесть риски (BR-011, бриф заказчика). Формат — **одно свободное текстовое поле** (Q 3.2); чекбоксы с фиксированным списком аллергенов **не используются**.

Секция встроена в поток бронирования: **контакты (SCR-013) → аллергии (SCR-012) → экипировка → подтверждение** ([domain-description.md §5.2](../../1-elicitation/domain-description.md#52-бронирование)). При **первой записи** шаг обязателен (Q 3.1); при **повторных** — подставляется сохранённое значение из профиля/последней брони, клиент может изменить или оставить как есть. В **существующей активной записи** аллергии редактируются отдельно (FR-013, Q 3.3).

При **несовместимости** указанных ограничений с меню класса приложение показывает **предупреждающий баннер**; запись **не блокируется** (FR-014, Q 3.4) — клиент принимает решение осознанно, студия видит данные на кухне.

### User Story

> Как клиент, я хочу указать аллергии и ограничения питания при записи на класс, чтобы шеф мог заранее учесть их на кухне.

**Не в MVP:** лист ожидания при `NO_SPOTS` (FR-011) — только возврат к расписанию; фильтр по шефу; iOS; чекбоксы фиксированных аллергенов; отдельный экран «Медицинская карта»; загрузка справок.

## Используемые запросы

### checkAllergyCompatibility

**Метод:** POST  
**Путь:** `/allergies/check`  
**Спецификация:** [../../api/openapi.yaml](../../api/openapi.yaml) → `checkAllergyCompatibility`

**Тело запроса:** `{ "slotId": "...", "allergiesText": "..." }`

**Обработка ответа:**

| HTTP / код | UI-реакция |
| :-- | :-- |
| 200 + `hasConflict: true` | Warning-баннер под полем с `message`; CTA «Записаться» и «Сохранить» **остаются активными** (FR-014) |
| 200 + `hasConflict: false` | Баннер скрыт |
| 404 | Баннер скрыт; запись возможна без проверки |
| 5xx / timeout | Баннер скрыт; запись не блокируется |
| Offline | Проверка пропускается; ввод и submit доступны |

### updateBookingAllergies

**Метод:** PATCH  
**Путь:** `/bookings/{bookingId}`  
**Спецификация:** [../../api/openapi.yaml](../../api/openapi.yaml) → `updateBookingAllergies`

**Тело запроса:** `{ "allergies": { "allergiesText": "...", "menuWarningAcknowledged": true } }`

**Обработка ответа:**

| HTTP / код | UI-реакция |
| :-- | :-- |
| 200 + data | Sheet закрывается, снек «Аллергии обновлены», блок на SCR-009 обновлён |
| 403 `BOOKING_NOT_ACTIVE` | «Изменение недоступно»; sheet закрывается |
| 401 | «Сессия истекла» |
| 5xx / timeout | Снек + «Повторить»; sheet остаётся открытым |
| Offline | Правка недоступна (NFR-009 — только просмотр кэша на SCR-009) |

### createBooking (аллергии в составе формы SCR-005)

**Метод:** POST  
**Путь:** `/bookings`  
**Спецификация:** [../../api/openapi.yaml](../../api/openapi.yaml) → `createBooking`

Аллергии передаются в теле: `allergies: { allergiesText, menuWarningAcknowledged }`. Профиль клиента обновляется на бэкенде для следующих визитов.

**Доменные коды createBooking:** `NO_SPOTS`, `ONE_BOOKING_PER_DAY`, `SLOT_CANCELLED`, `RENTAL_UNAVAILABLE`, `SLOT_REBOOK_FORBIDDEN` → SCR-007.

## Выбор паттерна: inline vs bottom sheet

| Вариант | Плюсы | Минусы |
| :-- | :-- | :-- |
| **Inline-секция на SCR-005** | Обязательный шаг **виден в потоке**; не требует лишнего тапа; согласован с SCR-013; проще заблокировать CTA до прохождения шага | Удлиняет скролл SCR-005; меньше места для длинного текста |
| **Bottom sheet на SCR-005** | Больше места для ввода; единый паттерн с SCR-002/003/010 | Обязательный шаг **легко пропустить** визуально; нужен явный gate («Указать аллергии») перед CTA |

### Рекомендация дизайнеру

**Основной паттерн — inline-секция на SCR-005** (между контактами и экипировкой): это единственный способ надёжно выполнить требование «обязательный шаг в приложении» (Q 3.5, FR-012) без лишней навигации.

**Bottom sheet** использовать:

- при тапе **«Изменить»** на повторной записи (компактная сводка на SCR-005);
- при редактировании из **SCR-009** (FR-013);
- при переходе по **push «запрос аллергий»** (deep link → SCR-009 → sheet SCR-012).

На SCR-005 при первой записи sheet **не открывается автоматически** — только inline.

## Элементы экрана

| Элемент | Описание | Источник данных | Валидация / поведение |
| :-- | :-- | :-- | :-- |
| Заголовок секции | «Аллергии и ограничения» | — | На SCR-005 пометка `*` при первой записи |
| Подсказка | «Укажите аллергии или напишите «нет», если ограничений нет» | — | Только при первой записи / пустом профиле |
| Текстовое поле | `TextArea`, многострочное, **свободный текст** | `allergiesText` | Max ~500 символов; без чекбоксов аллергенов (Q 3.2) |
| Placeholder | «Например: орехи, глютен, лактоза…» | — | — |
| Chip «Нет аллергий» | Быстрое заполнение значением «нет» | — | Опционально; не заменяет осознанный ввод; touch target ≥ 44 pt |
| Сводка (повтор) | 1–2 строки текста с ellipsis | `allergiesText` | Тап «Изменить» → sheet |
| Ссылка «Изменить» | На SCR-005 (повтор) и SCR-009 | — | Открывает bottom sheet |
| Баннер предупреждения | Жёлтый / warning, иконка ⚠ | `checkAllergyCompatibility.message` | FR-014: информирует, **не блокирует** CTA и «Сохранить» |
| Кнопка «Сохранить» | Только в sheet | — | Активна при изменениях; на SCR-009 — `updateBookingAllergies` |
| CTA «Записаться» | На SCR-005 | — | Disabled при первой записи, если поле пустое |
| Контекст класса (sheet) | Название и дата | `getSlot` / SCR-009 | Только в sheet; на SCR-005 не дублировать длинное меню |

**Не включать:** чекбоксы «глютен / лактоза / орехи», отдельный экран «Медицинская карта», загрузку справок, переключатель «есть аллергии / нет» вместо текста.

**Терминология:** **шеф**, **программа класса**, **тип кухни**; прокат (фартук, ножи) **не влияет на цену** (FR-015). Тон — «ограничения питания», не «диагноз»; студия — «шеф» / «на кухне», не «врач».

## Состояния экрана

| Состояние | Условие | Отображение |
| :-- | :-- | :-- |
| Loading | Загрузка `getProfile`/`getBooking` | Skeleton секции/блока |
| **Первая запись, пустой профиль** | `allergies.isSet === false` | Inline-поле развёрнуто; CTA SCR-005 disabled до непустого текста |
| **Первая запись, ввод** | Клиент печатает | Live-валидация длины; опционально debounced `checkAllergyCompatibility` |
| **Повторная запись** | `allergies.isSet === true` | Сводка + «Изменить»; CTA активен без повторного ввода |
| **Редактирование (sheet)** | Тап «Изменить» / push deep link | Sheet с текущим текстом и контекстом класса |
| **Предупреждение FR-014** | API: `hasConflict === true` | Баннер под полем; CTA / «Сохранить» **остаются активными** |
| **Нет конфликта** | API: совместимо или проверка не выполнена | Баннер скрыт |
| Submitting | `updateBookingAllergies` / `createBooking` в процессе | Loader на CTA; блокировка повторного тапа |
| **Успех (SCR-009)** | 200 OK | Sheet закрывается, снек «Аллергии обновлены», блок на SCR-009 обновлён |
| **Ошибка сети** | 5xx / timeout | Снек + «Повторить»; на SCR-009 правка недоступна офлайн |
| **Бронь не «Активна»** | Отменена / посещена | На SCR-009: read-only текст, ссылка «Изменить» **скрыта** |
| **Push без актуальной брони** | Класс прошёл / отменён | SCR-009 с пояснением; sheet не открывается автоматически |

> Сквозной паттерн — [LOGIC-008](../../5-mobile-app-spec/09_Логики/LOGIC-008_Паттерн-состояний-экрана.md).

## Сценарии и переходы

### UC-009 / первая запись (основной)

1. Клиент выбирает слот → SCR-005.
2. После контактов (SCR-013) отображается inline-секция SCR-012 с пустым полем.
3. Клиент вводит аллергии или тапает «Нет аллергий».
4. При вводе система (опционально) запрашивает `checkAllergyCompatibility` с меню класса.
5. При конфликте — баннер предупреждения (FR-014); клиент может продолжить.
6. Выбор экипировки → «Записаться» → `createBooking` с `allergies` + `menuWarningAcknowledged` при конфликте → SCR-006.

### Повторная запись

1. SCR-005 загружает сохранённые аллергии → сводка «Орехи, цитрусовые».
2. Клиент может сразу записаться или тапнуть «Изменить».
3. Sheet SCR-012 → правка → «Сохранить» → возврат на SCR-005 с обновлённой сводкой.
4. При конфликте с меню нового класса — баннер на SCR-005 (inline или в sheet).

### Изменение в существующей записи (FR-013)

1. SCR-008 → SCR-009, статус «Активна».
2. Тап «Изменить» в блоке аллергий → sheet SCR-012.
3. Клиент правит текст → «Сохранить» → `updateBookingAllergies`.
4. SCR-009 обновляет блок; кухня получает данные через бэкенд.

### Push «запрос аллергий» (FR-027)

1. Клиент получает push: «Укажите аллергии к классу „Паста и соусы“ 12 июля».
2. Тап → deep link на SCR-009 (`bookingId`).
3. Если бронь активна и поле пустое / устарело — **автооткрытие** sheet SCR-012 (один раз за сессию).
4. Сохранение через sheet; push **не заменяет** обязательный шаг при первой записи — это напоминание для уже созданной брони.

### Несовместимость с меню (альт. UC-002 §3a)

1. Клиент указал «орехи»; меню класса содержит орехи.
2. Под полем — баннер: «В меню класса есть орехи. Запись возможна — предупредите шефа на кухне».
3. Клиент нажимает «Записаться» — бронь **создаётся** (FR-014); в запросе `menuWarningAcknowledged: true`.
4. Тот же баннер виден на SCR-009 после записи, пока конфликт актуален.

## Критерии приёмки

| ID | Критерий |
| :-- | :-- |
| AC-001 | **Дано** первая запись и пустой профиль аллергий, **Когда** открыт SCR-005, **Тогда** inline-секция SCR-012 видна между контактами и экипировкой и CTA «Записаться» disabled при пустом поле (FR-012, Q 3.1). |
| AC-002 | **Дано** первая запись, **Когда** отображается секция аллергий, **Тогда** используется **одно** многострочное текстовое поле без чекбоксов фиксированных аллергенов (Q 3.2). |
| AC-003 | **Дано** повторная запись с сохранёнными аллергиями, **Когда** открыт SCR-005, **Тогда** показывается компактная сводка и ссылка «Изменить»; CTA «Записаться» активен без повторного ввода. |
| AC-004 | **Дано** `checkAllergyCompatibility` вернул `hasConflict=true`, **Когда** клиент продолжает оформление, **Тогда** показывается warning-баннер и при этом действия «Сохранить»/«Записаться» остаются активными (FR-014, Q 3.4). |
| AC-005 | **Дано** конфликт аллергий с меню и submit формы, **Когда** выполняется `createBooking`, **Тогда** в теле запроса передаётся `menuWarningAcknowledged: true` (LOGIC-009). |
| AC-006 | **Дано** активная запись на SCR-009, **Когда** клиент меняет аллергии и нажимает «Сохранить», **Тогда** выполняется `updateBookingAllergies` и блок SCR-009 обновляется по ответу 200 (FR-013). |
| AC-007 | **Дано** бронь не в статусе `ACTIVE`, **Когда** открыт SCR-009, **Тогда** аллергии отображаются read-only и ссылка «Изменить аллергии» **скрыта**. |
| AC-008 | **Дано** push «запрос аллергий» (FR-027) с `bookingId`, **Когда** клиент тапает уведомление, **Тогда** открывается SCR-009 и sheet SCR-012 (при активной брони). |
| AC-009 | **Дано** chip «Нет аллергий», **Когда** клиент на первой записи тапает chip, **Тогда** поле заполняется значением «нет» и CTA «Записаться» становится доступен после остальных обязательных шагов. |
| AC-010 | **Дано** конфликт исчез после смены меню на бэкенде, **Когда** пользователь повторно открывает SCR-005/009, **Тогда** warning-баннер снимается без участия клиента. |

## Заметки для дизайнера

- Секция должна восприниматься как **забота о безопасности на кухне**, а не как бюрократическая анкета — минимум элементов, тёплый тон (BR-011, бриф Артёма).
- Chip «Нет аллергий» снижает трение для клиентов без ограничений, но **не скрывает** сам шаг при первой записи.
- Баннер FR-014 — не красная ошибка и не modal: клиент **не должен** думать, что запись запрещена.
- На SCR-005 не дублировать длинное меню класса — только краткий контекст в sheet при редактировании с SCR-009.
- При deep link из push не показывать полный поток записи — только деталь брони + фокус на аллергиях.
- Если конфликт исчез после смены меню на бэкенде — баннер снимается при следующей загрузке SCR-005/009 без участия клиента.
- Поле доступно с screen reader: label, hint, озвучивание баннера предупреждения.
- Макеты: первая запись (inline), повторная (сводка), sheet редактирования, SCR-009 read/edit, состояние с баннером FR-014, пустое vs заполненное.
- Компонент «поле аллергий» включить в UI Kit ([design-brief.md](../design-brief.md)) для переиспользования на SCR-005 и SCR-009.

# Use Cases — кулинарная студия «Шеф-стол»

> Источник: [domain-description.md](../1-elicitation/domain-description.md).
> В скоупе текущей поставки — актор **Клиент** (R-028). Действия владельца и шефа выполняются в существующей инфраструктуре.
> Приоритет: **Must** / **Should**.

---

## UC-001. Просмотр расписания классов

| Поле | Описание |
| :-- | :-- |
| **ID** | UC-001 |
| **Название** | Просмотр расписания классов |
| **Актор** | Клиент |
| **Приоритет** | Must |
| **Источник** | [domain-description.md §5.1](../1-elicitation/domain-description.md#51-просмотр-расписания) |
| **Связь** | FR-001–FR-005; US-001–US-005 |

**Предусловия:** Клиент открыл клиентское приложение. Бэкенд доступен и отдаёт расписание.

**Основной сценарий:**
1. Система запрашивает слоты на ближайшие 7 дней (R-027).
2. Система отображает список классов: время, программа (кратко), шеф, рейтинг, свободные места, цена.
3. Клиент при необходимости применяет фильтры (время суток, уровень, тип кухни) или расширяет период через фильтр дат.
4. Клиент выбирает интересующий слот для дальнейших действий.

**Альтернативные сценарии:**
- **2a.** Слотов на ближайшие дни нет → система показывает empty state «Пока нет доступных классов».

## UC-002. Запись на кулинарный класс

| Поле | Описание |
| :-- | :-- |
| **ID** | UC-002 |
| **Название** | Запись на кулинарный класс |
| **Актор** | Клиент |
| **Приоритет** | Must |
| **Источник** | [domain-description.md §5.2](../1-elicitation/domain-description.md#52-бронирование) |
| **Связь** | FR-006–FR-015, FR-028; US-006–US-009, US-019, US-020 |

**Предусловия:** Клиент просматривает расписание (UC-001). В слоте есть свободные места. У клиента нет другой записи на этот день.

**Основной сценарий:**
1. Клиент выбирает слот.
2. Клиент указывает имя и телефон (при первой записи).
3. Клиент проходит обязательный шаг указания аллергий (текстовое поле).
4. Клиент выбирает: своё снаряжение или прокат (фартук, ножи).
5. Система отправляет запрос бронирования в бэкенд.
6. Бэкенд атомарно проверяет наличие мест и подтверждает бронь (R-004).
7. Система отображает клиенту подтверждение записи и цену (оплата на месте).

**Альтернативные сценарии:**
- **3a.** Аллергии несовместимы с меню → система показывает предупреждение; клиент может продолжить запись.
- **4a.** Прокатный фонд исчерпан → запись возможна только «со своим» снаряжением.
- **6a.** Мест нет или уже есть запись на этот день → бэкенд отклоняет запрос → система информирует клиента об отказе.
- **6b.** Группа заполнена → система отображает «мест нет».

## UC-003. Просмотр своих записей

| Поле | Описание |
| :-- | :-- |
| **ID** | UC-003 |
| **Название** | Просмотр своих записей |
| **Актор** | Клиент |
| **Приоритет** | Must |
| **Источник** | [domain-description.md §2, §5.3](../1-elicitation/domain-description.md#2-ключевые-сущности) |
| **Связь** | FR-016; US-010 |

**Предусловия:** У клиента есть одна или несколько броней в системе.

**Основной сценарий:**
1. Клиент открывает раздел своих записей.
2. Система отображает список броней со статусами (активна, отменена клиентом, отменена студией, посещена и др.).
3. Клиент просматривает детали выбранной записи.

**Альтернативные сценарии:**
- **2a.** Нет подключения к интернету → система показывает последний закэшированный список записей (NFR-009).

## UC-004. Отмена записи клиентом

| Поле | Описание |
| :-- | :-- |
| **ID** | UC-004 |
| **Название** | Отмена записи клиентом |
| **Актор** | Клиент |
| **Приоритет** | Must |
| **Источник** | [domain-description.md §3, §5.3](../1-elicitation/domain-description.md#53-отмена-брони) |
| **Связь** | FR-017, FR-018; US-011, US-012 |

**Предусловия:** У клиента есть активная бронь на предстоящий класс.

**Основной сценарий:**
1. Клиент открывает активную бронь.
2. Клиент инициирует отмену.
3. Если до начала ≥ 3 ч — система отменяет бронь и освобождает место в слоте.
4. Система обновляет статус брони на «отменена клиентом».

**Альтернативные сценарии:**
- **3a.** До начала < 3 ч → система показывает предупреждение о закупленных продуктах; клиент подтверждает отмену → бронь отменяется (штрафов в MVP нет).

## UC-005. Отмена класса студией (форс-мажор)

| Поле | Описание |
| :-- | :-- |
| **ID** | UC-005 |
| **Название** | Отмена класса студией (форс-мажор) |
| **Актор** | Клиент (получатель); инициатор — существующая инфраструктура (вне скоупа) |
| **Приоритет** | Must |
| **Источник** | [domain-description.md §3, §5.6](../1-elicitation/domain-description.md#56-форс-мажор) |
| **Связь** | FR-019–FR-022; US-013, US-014 |

**Предусловия:** Клиент записан на слот. Владелец/администратор отменяет класс в существующей инфраструктуре (срыв поставки, поломка оборудования и т. п.).

**Основной сценарий:**
1. Бэкенд переводит бронь в статус «Отменён студией» с указанием причины (R-008).
2. Клиентское приложение получает обновление статуса.
3. Система отправляет клиенту push-уведомление с просьбой перезаписаться на другой класс.
4. Клиент видит в приложении статус, причину и может перейти к перезаписи.

**Постусловия:** Повторная запись на тот же отменённый слот запрещена (R-008).

## UC-007. Оценка шефа

| Поле | Описание |
| :-- | :-- |
| **ID** | UC-007 |
| **Название** | Оценка шефа после класса |
| **Актор** | Клиент |
| **Приоритет** | Must |
| **Источник** | [domain-description.md §3, §5.5](../1-elicitation/domain-description.md#55-оценка-шефа) |
| **Связь** | FR-024–FR-026; US-016, US-017 |

**Предусловия:** Клиент посетил класс; бронь имеет статус «посещена». С момента класса прошло не более недели.

**Основной сценарий:**
1. Система предлагает клиенту оценить шефа (звёзды).
2. Клиент выставляет или изменяет оценку.
3. Система сохраняет оценку через API; рейтинг отображается другим клиентам при выборе слота.

**Постусловия:** У клиента — не более одной оценки на шефа.

## UC-008. Push-уведомления клиенту

| Поле | Описание |
| :-- | :-- |
| **ID** | UC-008 |
| **Название** | Push-уведомления клиенту |
| **Актор** | Клиент |
| **Приоритет** | Must |
| **Источник** | [domain-description.md §3, §5.4](../1-elicitation/domain-description.md#уведомления-mvp) |
| **Связь** | FR-027; US-018 |

**Предусловия:** У клиента есть активная бронь или завершённый сценарий записи/отмены.

**Основной сценарий:**
1. Система формирует push-уведомление по событию: напоминание за день, за 2 часа, подтверждение записи, отмена (клиент / студия), запрос аллергий.
2. Клиент получает push в приложении.

**Примечания:** Доставка только через push (NFR-010).

## UC-009. Управление аллергиями

| Поле | Описание |
| :-- | :-- |
| **ID** | UC-009 |
| **Название** | Указание и изменение аллергий |
| **Актор** | Клиент |
| **Приоритет** | Must |
| **Источник** | [domain-description.md §3, §5.2, §5.4](../1-elicitation/domain-description.md#аллергии) |
| **Связь** | FR-012–FR-014; US-007 |

**Предусловия:** Клиент оформляет новую запись или редактирует существующую.

**Основной сценарий:**
1. Клиент указывает аллергии в свободном текстовом поле (обязательный шаг при первой записи).
2. Система сохраняет данные; при повторной записи подставляет сохранённое значение.
3. При несовместимости с меню класса система показывает предупреждение.

**Альтернативные сценарии:**
- **1a.** Клиент изменяет аллергии в существующей записи → система обновляет данные брони.

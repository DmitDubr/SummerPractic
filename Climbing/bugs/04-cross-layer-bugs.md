# 04 — Сквозные расхождения (ТЗ ↔ Бэкенд ↔ Клиент)

Критичные сценарии, где слои расходятся друг с другом и с ТЗ.

---

## XL-01: Waitlist + бронь в один день

| Поле | Значение |
|------|----------|
| **ID** | XL-01 |
| **Приоритет** | P0 |
| **Статус** | Решён |
| **Связанные** | BE-01 |

| Слой | Поведение |
|------|-----------|
| **ТЗ** | Одна «запись» в день (бронь или очередь) |
| **Бэкенд** | Waitlist join блокирует; create booking — нет |
| **Клиент** | Не проверяет превентивно |

**Шаги воспроизведения:**

1. Клиент встаёт в очередь на слот в 10:00.
2. Клиент записывается на слот в 18:00 того же дня.
3. Ожидание: 409 `ONE_BOOKING_PER_DAY`.
4. Факт: 201 Created.

**Решение (2026-07-05):** BE-01 — `checkOneLiveBookingPerDay` в `CreateBooking`; тест `TestOneBookingPerDayAfterWaitlistIntegration`.

## XL-02: Граница 60 минут при отмене

| Поле | Значение |
|------|----------|
| **ID** | XL-02 |
| **Приоритет** | P0 |
| **Статус** | Решён |
| **Связанные** | BE-02 |

| Слой | Поведение |
|------|-----------|
| **ТЗ** | ≥60 мин → ранняя отмена, место сразу для waitlist |
| **Go** | `>= 1 hour` → notify waitlist |
| **PostgreSQL** | `> 1 hour` → release spot |
| **Клиент** | Показывает warning при `< 60 мин` (корректно для UI) |

**Риск:** рассинхрон waitlist notification и фактического освобождения места.

**Решение (2026-07-05):** BE-02 — граница `>= 1 hour` в Go и PostgreSQL (`000003`, `000005`).

## XL-03: Прокатный фонд — утечка при отменах

| Поле | Значение |
|------|----------|
| **ID** | XL-03 |
| **Приоритет** | P1 |
| **Статус** | Решён |
| **Связанные** | BE-03 |

| Слой | Поведение |
|------|-----------|
| **ТЗ** | Прокат атомарно резервируется и освобождается |
| **Бэкенд** | Reserve на create; нет release на cancel |
| **Клиент** | Корректно показывает `RENTAL_UNAVAILABLE` при submit |

**Эффект:** со временем слоты становятся «недоступны» без реальной нагрузки.

**Решение (2026-07-05):** BE-03 — восстановление проката в `trg_bookings_release_slot` (`000005`).

## XL-04: Waitlist lifecycle обрывается на клиенте

| Поле | Значение |
|------|----------|
| **ID** | XL-04 |
| **Приоритет** | P1 |
| **Статус** | Решён |
| **Связанные** | CMP-06, CMP-07, CMP-04 |

| Этап | Бэкенд | Клиент |
|------|--------|--------|
| Вступить в очередь | ✅ | ✅ |
| Показать позицию | ✅ API | ✅ |
| Покинуть очередь | ✅ API | ✅ |
| Push «место освободилось» | stub | ❌ deep link |
| Завершить запись после push | ✅ | ❌ |

**Решение (2026-07-05):** CMP-06/CMP-07 — позиция и выход из очереди на клиенте. Push/deep links остаются в CMP-04 (P2).

## XL-06: «Мои записи» не отражает жизненный цикл брони

| Поле | Значение |
|------|----------|
| **ID** | XL-06 |
| **Приоритет** | P1 |
| **Статус** | Решён |
| **Связанные** | CMP-05, CMP-09 |

| Слой | Поведение |
|------|-----------|
| **ТЗ** | Все статусы, upcoming + past, deep link target |
| **Бэкенд** | Отдаёт все статусы |
| **Клиент** | Показывает только Active + Waitlist |

**Следствие:** после отмены/посещения запись исчезает из UI вместо смены бейджа.

**Решение (2026-07-05):** CMP-05/CMP-09 — сегмент «Прошедшие», полная деталь записи.

# Use Cases — скалодром «Вертикаль»

> Источник: [domain-description.md](../1-elicitation/domain-description.md).
> В скоупе текущей поставки — актор **Клиент** (R-028). Действия владельца и инструктора выполняются в существующей инфраструктуре.
> Приоритет: **Must** / **Should**.

---

## UC-001. Просмотр расписания тренировок

| Поле | Описание |
| :-- | :-- |
| **ID** | UC-001 |
| **Название** | Просмотр расписания тренировок |
| **Актор** | Клиент |
| **Приоритет** | Must |
| **Источник** | [domain-description.md §5.1](../1-elicitation/domain-description.md#51-просмотр-расписания) |
| **Связь** | FR-001, FR-002, FR-003, FR-004; US-001–US-004 |

**Предусловия:** Клиент открыл клиентское приложение. Бэкенд доступен и отдаёт расписание.

**Основной сценарий:**
1. Система запрашивает слоты на ближайшие 7 дней (R-027).
2. Система отображает список слотов: время начала, зона/формат, инструктор, свободные места.
3. Клиент при необходимости расширяет период через фильтр дат.
4. Клиент выбирает интересующий слот для дальнейших действий.

**Альтернативные сценарии:**
- **4a.** Слотов на ближайшие дни нет → система показывает empty state «Пока нет доступных тренировок».

## UC-002. Запись на тренировку

| Поле | Описание |
| :-- | :-- |
| **ID** | UC-002 |
| **Название** | Запись на тренировку |
| **Актор** | Клиент |
| **Приоритет** | Must |
| **Источник** | [domain-description.md §5.2](../1-elicitation/domain-description.md#52-бронирование) |
| **Связь** | FR-005, FR-006; US-005, US-006 |

**Предусловия:** Клиент просматривает расписание (UC-001). В слоте есть свободные места.

**Основной сценарий:**
1. Клиент выбирает слот.
2. Клиент указывает: своё снаряжение или прокат (скальники, страховочная система).
3. Система отправляет запрос бронирования в бэкенд.
4. Бэкенд атомарно проверяет наличие мест и подтверждает бронь (R-004).
5. Система отображает клиенту подтверждение записи.

**Альтернативные сценарии:**
- **4a.** Мест нет → бэкенд отклоняет запрос → система информирует клиента об отказе.

| **Связь** | FR-007; US-007 |

**Предусловия:** У клиента есть одна или несколько броней в системе.

2. Система отображает список броней со статусами (активна, отменена клиентом, отменена скалодромом, посещена и др.).
3. Клиент просматривает детали выбранной записи.

## UC-004. Отмена записи клиентом

| Поле | Описание |
| :-- | :-- |
| **ID** | UC-004 |
| **Название** | Отмена записи клиентом |
| **Актор** | Клиент |
| **Приоритет** | Must |
| **Источник** | [domain-description.md §5.3](../1-elicitation/domain-description.md#53-отмена-брони) |
| **Связь** | FR-008; US-008 |

**Предусловия:** У клиента есть активная бронь на предстоящую тренировку.

**Основной сценарий:**
1. Клиент открывает активную бронь.
2. Клиент инициирует отмену.
3. Система отменяет бронь и освобождает место в слоте (при заблаговременной отмене).
4. Система обновляет статус брони на «отменена клиентом».

**Примечания:** Правила поздней отмены (около 10 минут до начала) в домене отмечены как требующие уточнения — в UC не детализируются.

## UC-005. Отмена тренировки скалодромом

| Поле | Описание |
| :-- | :-- |
| **ID** | UC-005 |
| **Название** | Отмена тренировки скалодромом (профилактика) |
| **Актор** | Клиент (получатель); инициатор — существующая инфраструктура (вне скоупа) |
| **Приоритет** | Must |
| **Источник** | [domain-description.md §3, §5.6](../1-elicitation/domain-description.md#36-профилактика-зоны) |
| **Связь** | FR-009, FR-010, FR-011; US-009, US-010 |

**Предусловия:** Клиент записан на слот. Владелец/администратор отменяет тренировку в существующей инфраструктуре (профилактика зоны, перекрутка трасс, проверка снаряжения).

**Основной сценарий:**
1. Бэкенд переводит бронь в статус «Отменена скалодромом» с указанием причины (R-008).
2. Клиентское приложение получает обновление статуса.
3. Система отправляет клиенту push-уведомление об отмене.
4. Клиент видит в приложении статус, причину и детали отменённой тренировки.

**Постусловия:** Повторная запись на тот же отменённый слот запрещена (R-008).

| Поле | Описание |
| :-- | :-- |
| **ID** | UC-006 |
| **Название** | Оценка инструктора после тренировки |
| **Актор** | Клиент |
| **Приоритет** | Should |

| **Связь** | FR-012; US-011 |

**Предусловия:** Клиент посетил тренировку; бронь имеет статус «посещена».

**Основной сценарий:**

2. Клиент выставляет оценку.
3. Система сохраняет оценку через API.

**Постусловия:** Оценка доступна для аналитики качества работы инструкторов.

| Поле | Описание |
| :-- | :-- |
| **ID** | UC-007 |
| **Название** | Push-напоминание о предстоящей тренировке |

| **Связь** | FR-013; US-012 |

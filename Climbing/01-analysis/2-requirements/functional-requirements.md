# Функциональные требования — скалодром «Вертикаль»

> Источник: [domain-description.md](../1-elicitation/domain-description.md).
> ФТ описывают **что** делает система. Нефункциональные аспекты — в [non-functional-requirements.md](non-functional-requirements.md).
> Приоритет: **Must** / **Should** / **Could**.

| ID | Формулировка | Приоритет | Источник |
| :- | :-- | :--: | :-- |
| FR-001 | Клиентское приложение отображает список доступных слотов на **ближайшие 7 дней** от текущего момента (R-027). | Must | [domain-description.md §3, §5.1](../1-elicitation/domain-description.md#51-просмотр-расписания) |
| FR-002 | Клиент может **расширить период** просмотра расписания с помощью фильтра дат. | Must | [domain-description.md §5.1](../1-elicitation/domain-description.md#51-просмотр-расписания) |
| FR-003 | Для каждого слота отображаются: **время начала**, **зона/формат**, **инструктор**, **количество свободных мест**. | Must | [domain-description.md §5.1](../1-elicitation/domain-description.md#51-просмотр-расписания) |
| FR-004 | При отсутствии доступных слотов на ближайшие дни приложение показывает empty state: **«Пока нет доступных тренировок»**. | Must | [domain-description.md §5.1, §8](../1-elicitation/domain-description.md#8-глоссарий) |
| FR-005 | Клиент может **записаться на выбранный слот**, указав, приходит ли со **своим снаряжением** или нужен **прокат** (скальники, страховочная система). | Must | [domain-description.md §5.2](../1-elicitation/domain-description.md#52-бронирование) |
| FR-006 | При бронировании приложение отправляет запрос существующему бэкенду и **отображает результат**: подтверждение брони или отказ при отсутствии свободных мест (R-004). | Must | [domain-description.md §3, §5.2](../1-elicitation/domain-description.md#52-бронирование) |
| FR-007 | Клиент может **просматривать свои брони** (активные и завершённые) со статусами: активна, отменена клиентом, отменена скалодромом, посещена и др. | Must | [domain-description.md §2](../1-elicitation/domain-description.md#2-ключевые-сущности) |
| FR-008 | Клиент может **отменить свою бронь**; при заблаговременной отмене место в слоте освобождается. | Must | [domain-description.md §3, §5.3](../1-elicitation/domain-description.md#53-отмена-брони) |
| FR-009 | При отмене тренировки **скалодромом** (профилактика, закрытие зоны) бронь сохраняется со статусом **«Отменена скалодромом»** и **причиной отмены** (R-008). | Must | [domain-description.md §3, §5.3, §5.6](../1-elicitation/domain-description.md#36-профилактика-зоны) |
| FR-010 | Клиенту отправляется **push-уведомление** при отмене тренировки скалодромом (R-008). | Must | [domain-description.md §3](../1-elicitation/domain-description.md#3-бизнес-правила) |
| FR-011 | Повторная запись клиента на **тот же отменённый скалодромом слот** запрещена (R-008). | Must | [domain-description.md §3](../1-elicitation/domain-description.md#3-бизнес-правила) |
| FR-012 | После **посещённой** тренировки клиент может **оценить инструктора**. | Should | [domain-description.md §3, §5.5](../1-elicitation/domain-description.md#55-оценка-инструктора) |
| FR-013 | Приложение отправляет клиенту **push-напоминания** о предстоящих тренировках для снижения неявок. | Should | [domain-description.md §5.4](../1-elicitation/domain-description.md#54-напоминания) |
| FR-014 | Слоты, зоны/форматы и данные инструкторов **поступают из существующего бэкенда** через API; приложение их потребляет, но **не создаёт и не редактирует**. | Must | [domain-description.md §1, §6](../1-elicitation/domain-description.md#6-границы-системы) |
| FR-015 | Клиентское приложение реализует сценарии **только для роли «Клиент»** (R-028). | Must | [domain-description.md §4, §6](../1-elicitation/domain-description.md#4-акторы) |
| FR-016 | API клиентского приложения обеспечивает операции по контракту: **auth**, **слоты**, **бронирования**, **профиль**, **инструкторы/рейтинги**. | Must | [domain-description.md §6](../1-elicitation/domain-description.md#6-границы-системы) |
